## 需求摘要
使用者希望能自助式地建立 ETL 排程、拉出報表、組成自訂 Dashboard。核心功能是三層架構：
1. **ETL 層**：使用者在網頁上設定資料抽取 → 轉換 → 載入的排程
2. **BI 報表層**：使用者拖拉製作 chart 和 table
3. **Dashboard 層**：多個 chart/table 組合成 dashboard，可儲存、命名、分享

## 已確認的設計決策
- 利用現有的 Connector Manager（DB 連線）和 Sync Engine（資料同步）
- ETL 排程用 Celery 執行，跟現有 batch sync 共用基礎設施
- BI 功能用前端 ECharts 渲染，後端提供資料查詢 API

## 使用者故事

### 大數據工程師（Big Data Engineer）
- **作為**大數據工程師，**我想要**在網頁上建立 ETL pipeline（來源表 → 轉換規則 → 目標表），**以便**自動化資料整理而不用寫程式
- **驗收條件**：
  - [ ] 可選擇來源資料庫和表（從已連線的 DB 中選）
  - [ ] 可設定轉換規則：欄位篩選、重命名、型別轉換、簡單運算、JOIN
  - [ ] 可設定目標表（寫入 platform DB 或指定的資料庫）
  - [ ] 可設定排程（每 N 分鐘 / 每小時 / 每日 / 自訂 cron）
  - [ ] 可手動觸發執行
  - [ ] 執行歷史和狀態可查詢
  - [ ] 失敗自動重試 + 告警

- **作為**大數據工程師，**我想要**管理所有 ETL pipeline 的生命週期，**以便**維持資料流的健康
- **驗收條件**：
  - [ ] Pipeline 列表顯示：名稱、來源、目標、排程、上次執行狀態、下次執行時間
  - [ ] 可啟用/停用 pipeline
  - [ ] 可查看執行日誌（耗時、處理筆數、錯誤）
  - [ ] Pipeline 可版本控制（修改前後可比較）

### 各部門使用者（所有角色）
- **作為**有權限的使用者，**我想要**從資料表中選擇資料，用拖拉方式做出 chart 或 table，**以便**自己做分析報表不用等大數據部門
- **驗收條件**：
  - [ ] 可選擇資料表（或 ETL 產出的表）作為資料來源
  - [ ] 可選擇欄位作為 X 軸、Y 軸、分組依據
  - [ ] 支援圖表類型：折線圖、柱狀圖、圓餅圖、面積圖、散點圖
  - [ ] 支援 table 顯示：可排序、篩選、分頁
  - [ ] 可設定篩選條件（日期範圍、特定值）
  - [ ] 圖表可儲存為「報表元件」（reusable）

- **作為**有權限的使用者，**我想要**把多個 chart 和 table 組合成一個 dashboard 頁面並固定下來，**以便**每次打開就能看到我關心的指標
- **驗收條件**：
  - [ ] 可建立空白 dashboard
  - [ ] 可從已儲存的報表元件中拖拉加入 dashboard
  - [ ] 元件可自由排列和調整大小（grid layout）
  - [ ] Dashboard 可命名和儲存
  - [ ] Dashboard 可分享給其他使用者（唯讀或可編輯）
  - [ ] 支援自動刷新（設定刷新頻率）

- **作為**有權限的使用者，**我想要**匯出報表和 dashboard，**以便**在會議中使用
- **驗收條件**：
  - [ ] 單一 chart/table 可匯出為 PNG / Excel
  - [ ] 整個 dashboard 可匯出為 PDF
  - [ ] 支援定時寄送報表到 email（排程）

### PM（Project Manager）
- **作為** PM，**我想要**建立自己的專案追蹤 dashboard，**以便**不用每次都從多個頁面拼湊資訊
- **驗收條件**：
  - [ ] 可從 MRP、庫存、品質、排程等資料表拉出元件
  - [ ] Dashboard 開啟後一目了然看到專案全貌

### MIS 系統管理員（MIS Administrator）
- **作為** MIS 管理員，**我想要**控制誰可以建立 ETL pipeline 和 dashboard，**以便**防止使用者做出影響效能的查詢
- **驗收條件**：
  - [ ] ETL 建立權限可依角色控制（預設只有 big_data 和 mis）
  - [ ] Dashboard 建立權限可更廣泛（但查詢範圍受 RBAC 限制）
  - [ ] 查詢有 timeout 限制（防止超大查詢拖垮系統）
  - [ ] ETL 執行的資源使用可監控

## 衝突標記
- ⚠️ **各部門使用者**想要自由查詢任意資料表，但 **MIS 管理員**擔心效能和安全性 → 見 conflicts/CONFLICT-005.md

## 非功能需求
- 效能：查詢 timeout 上限 30 秒；ETL 單次執行上限 10 分鐘
- 安全性：BI 查詢受 RBAC 限制（使用者只能查詢有權限的資料表）
- 可用性：chart builder 用拖拉介面，不需要寫 SQL
- 儲存：Dashboard 定義儲存在 platform DB（JSON 格式）
