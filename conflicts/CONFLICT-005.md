# 衝突編號：CONFLICT-005

## 狀態：`detected`

## 衝突描述
- **各部門使用者** 需要：自由查詢資料表來建立 BI 報表和 dashboard，才能自助式做分析
- **MIS 管理員** 需要：防止使用者做出超大查詢（如 SELECT * 全表掃描）拖垮資料庫效能
- **為什麼衝突**：BI 自助查詢的價值在於「自由度」，但不受限的查詢可能產生效能問題。特別是 Tiptop Oracle 資料庫，一個大查詢就可能影響 ERP 正常使用。

## AI 分析
這是之前 CONFLICT-002（即時查詢 vs ERP 效能）的延伸。BI 報表的查詢通常比簡單的庫存查詢更複雜（JOIN、GROUP BY、大範圍日期），風險更高。

核心問題：
- 使用者不懂 SQL，拖拉出來的查詢可能很低效
- 查詢的資料量可能很大（百萬筆以上）
- 直接對 Tiptop 做 BI 查詢 vs 對同步後的副本做查詢，效能影響完全不同

## 解決方案
- [ ] 方案 1：（優先開放）使用者可自由查詢，但系統自動加上 timeout（30 秒）和資料量上限（10 萬筆）
- [ ] 方案 2：（優先管控）BI 查詢只能對 platform DB 中的同步副本做，不能直接查 Tiptop
- [ ] 方案 3：（折衷方案）BI 查詢預設只對同步副本做（快且安全）。進階使用者（big_data 角色）可以選擇直接查來源 DB，但有 timeout + 資源限制 + audit log
