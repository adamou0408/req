# 此框架與 GitHub Spec Kit 的詳細比較分析

## 背景

**GitHub Spec Kit**（簡稱 speckit）是 GitHub 於 2025 年 9 月發布的開源工具，推動「規格驅動開發」（Spec-Driven Development, SDD）方法論。目前在 GitHub 上擁有約 85,000+ stars，是 SDD 運動的旗艦工具。

**此框架**（需求驅動設計開發）是一個獨立建構在 Claude Code 之上的閉環開發系統。

---

## 1. 核心理念對比

| 面向 | 此框架 | Spec Kit |
|------|--------|----------|
| **核心哲學** | 規格是唯一真相來源，程式碼是規格的衍生物 | 規格是唯一真相來源，程式碼是規格的衍生物 |
| **方法論名稱** | 需求驅動設計開發（Demand-Driven） | 規格驅動開發（Spec-Driven Development） |
| **目標受眾** | 非技術人員也能提需求（零門檻） | 開發者/技術團隊 |
| **輸入格式** | **任何格式**（口語、截圖、會議記錄、聊天訊息） | 結構化的規格描述（需開發者撰寫） |
| **AI 角色** | AI 負責翻譯、分析、生成；人類守門 | AI 負責生成程式碼；人類撰寫規格 |

**關鍵差異**：Spec Kit 假設**開發者**會撰寫結構化規格；此框架假設**任何人**都能用任何形式輸入，AI 負責結構化。

## 2. 命令/工作流對比

| 此框架命令 | Spec Kit 命令 | 功能對比 |
|-----------|-------------|---------|
| `/intake` | *(無對應)* | 原始需求收集，接受任何格式 |
| `/translate` | `/speckit.specify` | 生成結構化規格 |
| `/detect-conflicts` | *(無對應)* | 角色間衝突偵測 |
| `/review` | *(無對應)* | 人工審核閘門 |
| `/plan` | `/speckit.plan` | 生成技術架構計畫 |
| `/implement` | `/speckit.implement` | AI 生成程式碼與測試 |
| `/deploy` | *(無對應)* | 自動化部署與健康檢查 |
| `/feedback` | *(無對應)* | 生產環境監控回饋迴路 |
| `/iterate` | *(無對應)* | 需求變更管理 |
| *(無對應)* | `/speckit.constitution` | 建立專案治理原則 |
| *(無對應)* | `/speckit.tasks` | 產生任務清單（此框架內建於 `/plan`） |

**觀察**：此框架有 **9 個命令**，Spec Kit 有 **5 個命令**。此框架覆蓋了完整的軟體生命週期（含部署、監控、回饋），而 Spec Kit 專注於「規格→程式碼」的轉換。

## 3. 架構深度對比

### 規格結構

**此框架** (`specs/{feature}/spec.md`)：
```markdown
## 狀態：draft | in-review | approved | in-progress | done
## 來源追溯（連結到 intake/raw/）
## 使用者故事（按角色分組，含驗收條件）
## 衝突標記（連結到 conflicts/CONFLICT-XXX.md）
## 非功能需求
## 開放問題
```

**Spec Kit** (`.specify/specs/{branch}/`)：
```markdown
spec.md        — 功能描述（what & why）
plan.md        — 技術選擇與架構
tasks/         — 可執行任務清單
research.md    — 技術調研
contracts/     — API 合約與資料模型
```

**差異**：此框架的規格包含**角色衝突追蹤**、**狀態機**、**雙向溯源**；Spec Kit 的規格更簡潔，專注於技術描述，額外提供 `research.md`（技術調研）和 `contracts/`（API 合約）。

### 狀態管理

**此框架**：完整的狀態機
```
draft → in-review → approved → in-progress → done
                  ↘ draft（退回）
approved/in-progress/done → draft（透過 /iterate 變更需求）
```
加上衝突狀態：`detected → under-discussion → resolved`
加上任務狀態：`todo → in-progress → done → needs-human-intervention`

**Spec Kit**：無明確的狀態機定義，由開發者自行管理規格的演進。狀態隱含在 Markdown 成品中，無正式的轉換規則。

### 目錄結構

**此框架**（6 個頂層關注分離目錄）：
```
intake/          ← 不可變的原始記錄（append-only）
personas/        ← 角色定義庫
specs/           ← spec.md + plan.md + tasks.md
conflicts/       ← 獨立的衝突記錄
reviews/         ← 獨立的審核清單
infra/           ← 基礎設施即程式碼
```

**Spec Kit**（收斂在單一目錄）：
```
.specify/memory/              ← constitution.md
.specify/specs/{branch}/      ← spec.md + plan.md + tasks/ + research.md + contracts/
```

## 4. Constitution 模式對比

| 面向 | 此框架 `CONSTITUTION.md` | Spec Kit `/speckit.constitution` |
|------|--------------------------|----------------------------------|
| **性質** | 專案層級的不可違反原則（6 條） | 技術偏好的治理條款（如 Library-First、Test-First） |
| **關注點** | 流程治理（誰能決策、什麼不可跳過） | 技術治理（優先用套件、先寫測試、反過度抽象） |
| **範圍** | 覆蓋整個生命週期（intake→deploy→feedback） | 僅覆蓋程式碼產出階段 |
| **語氣** | 組織治理文件 | 技術團隊協議 |

---

## 5. Agent 行為規範對比

| 面向 | 此框架 `AGENTS.md` | Spec Kit `AGENTS.md` |
|------|---------------------|----------------------|
| **目的** | 定義 AI 的 MUST/MUST NOT 約束（9 大指令） | 定義各 AI 後端的整合設定（設定檔） |
| **性質** | **行為合約**（AI 不能做什麼） | **整合配置**（AI 如何接收指令） |
| **關鍵約束** | 不能替人決策、不能跳過狀態、不能部署未批准的規格 | 無行為約束，僅描述各 agent 的目錄結構與指令格式 |

核心差異：此框架**不信任 AI 的判斷力**（所以設立嚴格約束），Spec Kit **信任開發者的判斷力**（所以只提供工具）。

## 6. 此框架獨有的能力（Spec Kit 不具備）

| 能力 | 說明 |
|------|------|
| **零門檻輸入** | 接受任何格式，AI 負責結構化 |
| **角色衝突偵測** | 自動偵測不同 persona 間的需求衝突（4 種類型：功能、優先級、權限、UX） |
| **人工審核閘門** | 強制的人工批准流程，含完整審核清單 |
| **閉環回饋** | 生產問題自動變成新需求（monitor → intake） |
| **自動部署** | 含健康檢查、自動回滾、三環境管理（dev/staging/prod） |
| **需求變更管理** | `/iterate` 支援需求變更的影響分析與狀態回退 |
| **完整溯源鏈** | intake → spec → plan → tasks → code → deploy，CI 強制驗證 |
| **不可變歷史** | intake/raw/ 一旦提交即不可修改，作為永久記錄 |
| **AI 行為約束** | AGENTS.md 定義 9 大 MUST/MUST NOT 指令 |
| **CI Spec Gate** | 程式碼無法合併，除非有對應的已批准規格 |
| **監控告警去重** | 30 分鐘內相同告警自動合併，防止 intake 洪水 |
| **三次自動修復** | 測試失敗時自動修復最多 3 次，之後才交給人工 |

## 8. 工作流嚴格程度對比

| 面向 | 此框架 | Spec Kit |
|------|--------|----------|
| **階段數** | 10 個 | 5 個 |

## 9. 思考分工對比

```
Spec Kit 流程中的思考分工：
  人類想清楚要什麼 → 人類用 /speckit.specify 描述 → AI 細化 → 
  人類確認 → AI 規劃 → AI 產生任務 → AI 實作
  （人類負責：定義需求 + 確認規格）

此框架流程中的思考分工：
  人類隨便說一句話 → AI 理解意圖（/translate）→ AI 發現衝突 →
  人類只需「選 A 或選 B」→ 人類按「批准」→ AI 規劃 → AI 實作 →
  AI 部署 → AI 監控 → AI 自動報告問題 → 回到起點
  （人類負責：做選擇題 + 按批准鈕）
```

## 11. 結論

### 有關聯嗎？

**思想層面**：是的。兩者屬於同一場「規格驅動開發」運動，共享核心信念——**規格先於程式碼**。

**實作層面**：否。

- **無程式碼依賴**：此框架未引用任何 Spec Kit 的套件或程式碼
- **無直接致敬**：git 歷史中無任何 speckit 提及
- **範圍差異巨大**：Spec Kit 是「規格→程式碼」工具；此框架是「需求→部署→監控→回饋」的完整生命週期系統
- **受眾不同**：Spec Kit 面向開發者；此框架面向包括非技術人員在內的所有利害關係人

### 如何定位此框架？

此框架可以被視為 Spec Kit 理念的**超集擴展**——它不僅涵蓋了 Spec Kit 的核心功能（規格→計畫→實作），還向上游（需求收集、衝突偵測、人工審核）和下游（部署、監控、回饋迴路）大幅延伸，形成了一個完整的閉環系統。

一句話總結：**Spec Kit 是給開發者的「規格→程式碼」轉換工具；此框架是給整個組織的「需求→交付→回饋」閉環系統。**
