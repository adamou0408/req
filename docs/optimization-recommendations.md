# 框架優化建議：與市場主流 SDD 工具的差距分析

基於與 GitHub Spec Kit（85K+ stars）、AWS Kiro（GA 2025/11）、Tessl、Cursor Rules 等主流工具的比較，以及對本框架的深度審查，整理以下優化建議。

---

## 優先級分類

- **P0（關鍵）**：不解決會在生產環境造成失敗
- **P1（重要）**：不解決會造成明顯摩擦與返工
- **P2（建議）**：提升開發體驗與可擴展性

---

## P0 — 關鍵缺口

### 1. 新增 `/research` 階段（調研/去重）

**問題**：`/intake` 直接跳到 `/translate`，沒有調研階段。兩個人可以提交幾乎相同的需求，產生兩份重複的 spec。

**其他工具怎麼做**：
- Spec Kit 有 `research.md`，在規劃前先進行技術調研
- Kiro 有 "steering" 階段，先理解上下文再生成規格
- Tessl 有自動去重與需求聚合

**建議**：在 `/translate` 前新增 `/research` 命令：
```
/intake → /research → /translate → /detect-conflicts → ...
```

`/research` 應該做：
- 搜索已有 specs，檢查是否有重複或重疊需求
- 進行可行性初步評估
- 調查相關技術選項
- 產出 `specs/{feature}/research.md`
- 如果發現重複，建議合併而非新建

**新增檔案**：
- `.claude/commands/research.md`
- `specs/_template/research.md`

---

### 2. AI 上下文管理策略

**問題**：框架未定義 AI 的 token/上下文限制處理策略。當 spec 有 15 個 User Story、plan 有 20 個 tasks 時，AI 可能靜默截斷上下文，遺漏需求。

**其他工具怎麼做**：
- Spec Kit 明確提出「意圖壓縮」（Intentional Compaction）—— 每個階段將上下文壓縮為一份精簡文件，確保下一階段只接收必要資訊
- Kiro 有上下文裁剪機制，根據當前任務動態選擇相關片段

**建議**：
1. 在 `AGENTS.md` 新增「上下文管理」指令：
   ```
   ### 10. 上下文管理
   - 每個命令執行前，MUST 評估輸入文件的大小
   - 若 spec.md 超過 3000 字，MUST 產出摘要版本供下游使用
   - /plan 讀取 spec 時，MUST 只提取 User Stories 和驗收條件
   - /implement 執行每個 task 時，MUST 只載入該 task 相關的 spec 片段
   - MUST NOT 將整個 spec + plan + tasks 同時載入上下文
   ```

2. 在每個 stage 的 output 中加入「摘要區塊」：
   ```markdown
   ## 下游摘要（供下一階段使用）
   - 核心需求：3 句話概述
   - 關鍵約束：列出最重要的 3 個
   - 風險項目：列出最高風險的 2 個
   ```

---

### 3. 衝突解決流程（`/resolve-conflict`）

**問題**：框架偵測衝突但沒有解決流程。`conflicts/CONFLICT-XXX.md` 有 3 個選項，但沒有工具引導人類做決策，衝突容易卡在 `detected` 狀態。

**建議**：新增 `/resolve-conflict` 命令：
```
/resolve-conflict [conflict path]
```
行為：
1. 讀取衝突記錄，呈現 AI 分析摘要
2. 提供結構化的決策框架：
   - 影響範圍：各方案影響哪些 persona？
   - 成本估算：各方案的開發成本差異
   - 風險評估：各方案的潛在風險
3. 要求人類選擇方案並填寫理由
4. 更新衝突狀態為 `resolved`
5. 更新對應 spec.md 的衝突標記
6. 記錄到 changelog

**新增檔案**：`.claude/commands/resolve-conflict.md`

---

### 4. 安全性與合規

**問題**：框架幾乎沒有安全性考量。無密鑰管理、無依賴掃描、無 SBOM。

**其他工具怎麼做**：
- Kiro 內建安全掃描
- Spec Kit 社群有 `run_secret_scanning` 整合
- 業界標準要求 SBOM、依賴掃描、secrets scanning

**建議**：
1. 在 CI pipeline 加入：
   - `gitleaks` 或 GitHub secret scanning
   - `trivy` 容器漏洞掃描
   - 依賴項安全檢查（`npm audit` / `pip audit`）

2. 在 spec template 加入安全區塊：
   ```markdown
   ## 安全性需求
   - 資料分類：（公開 / 內部 / 機密 / 高度機密）
   - 認證需求：
   - 授權需求：
   - 加密需求：
   - 審計日誌需求：
   ```

3. 在 `CONSTITUTION.md` 加入第 7 條原則：
   ```markdown
   ## Principle 7: Security by Default
   > 安全性不是功能，而是基礎。
   - 每個 spec 必須包含安全性需求評估
   - 生成的程式碼必須通過安全掃描
   - 密鑰絕不進入版本控制
   - 依賴項必須定期掃描漏洞
   ```

---

### 5. 資料庫遷移策略

**問題**：框架完全沒有提及資料庫 schema 變更。部署新版本時如果有 schema 變更，rollback 可能導致資料遺失。

**建議**：
1. 在 plan template 加入：
   ```markdown
   ## 資料模型變更
   - 新增表格/欄位：
   - 修改表格/欄位：
   - 刪除表格/欄位：
   - 遷移策略：（向前相容 / 藍綠部署 / 多階段遷移）
   - 回滾策略：（此遷移是否可逆？）
   ```

2. 在 `/deploy` 中加入遷移檢查：
   - 部署前驗證遷移腳本存在
   - 回滾前驗證遷移可逆
   - 不可逆遷移需要額外人工批准

---

### 6. 程式碼審查流程

**問題**：AI 生成的程式碼直接進入部署流程，沒有人工程式碼審查環節。

**其他工具怎麼做**：
- Spec Kit：程式碼產出後由開發者審查
- Kiro：內建 PR review 工作流
- 業界標準：至少需要一個人類審查者

**建議**：
1. 在 CI pipeline 的 spec gate 之後加入 code review gate
2. 在 `AGENTS.md` 加入：
   ```
   ### 10. 程式碼審查
   - AI 生成的程式碼 MUST 經過人工審查後才能合併
   - 審查重點：安全性、效能、可維護性
   - 可選擇性跳過：僅限 hotfix 且需記錄原因
   ```
3. 考慮在 `/implement` 完成後自動建立 PR，而非直接合併

## P1 — 重要改進

### 7. Spec 版本管理

**問題**：Spec 只有狀態（draft/approved/done），沒有版本號。當 `/iterate` 多次修改同一 spec 後，無法知道「部署的是 v3 還是 v5」。

**建議**：在 spec template 加入版本追蹤：
```markdown
## 版本歷史
| 版本 | 日期 | 變更摘要 | 觸發者 |
|------|------|----------|--------|
| v1.0 | 2026-04-01 | 初始版本 | /translate |
| v1.1 | 2026-04-03 | 增加管理員角色 | /iterate |
| v2.0 | 2026-04-05 | 重新定義搜尋範圍 | /iterate |
```

---

### 8. Spec 依賴圖

**問題**：Spec 之間可能有依賴關係（如「認證」必須在「授權」之前完成），但框架不追蹤。

**其他工具怎麼做**：
- Kiro 有「hooks」系統追蹤 spec 間的依賴
- Tessl 有自動依賴分析

**建議**：在 spec template 加入：
```markdown
## 依賴關係
- 前置需求：[連結到必須先完成的 spec]
- 後續需求：[連結到依賴此 spec 的其他 spec]
- 共享資源：[列出與其他 spec 共用的元件]
```
在 `/plan` 中驗證前置需求的狀態。

---

### 9. API 合約與資料模型

**問題**：Spec Kit 有 `contracts/` 目錄存放 API 合約和資料模型，此框架沒有。

**建議**：在 spec 結構中加入：
```
specs/{feature-slug}/
  ├── spec.md
  ├── research.md      ← 新增（P0 #1）
  ├── contracts/        ← 新增
  │   ├── api.md       （API 端點定義）
  │   └── data-model.md（資料模型定義）
  ├── plan.md
  └── tasks.md
```

---

### 10. 測試策略分層

**問題**：框架只說「生成測試」，沒有區分測試類型。

**其他工具怎麼做**：
- Spec Kit 有 `Test-First Imperative`，明確要求先寫測試
- Kiro 有分層測試策略（unit → integration → e2e）

**建議**：在 tasks template 中為每個 task 標註測試層級：
```markdown
### 任務 1：使用者搜尋 API
- **測試策略**：
  - Unit: 搜尋邏輯函數
  - Integration: API → 資料庫查詢
  - E2E: 前端搜尋框 → API → 結果顯示
```

---

### 11. 並行任務標記

**問題**：tasks.md 中的任務按依賴順序排列，但沒有標記哪些可以並行執行。

**其他工具怎麼做**：
- Spec Kit 用 `[P]` 標記可並行的任務

**建議**：在 tasks template 加入並行標記：
```markdown
### 任務 1：建立 API 端點 [P-group-A]
### 任務 2：建立前端元件 [P-group-A]  ← 可與任務 1 並行
### 任務 3：整合測試 [depends: 1, 2]  ← 必須等 1 和 2 完成
```

---

### 12. 框架自身的可觀測性

**問題**：框架監控部署的應用，但不監控框架本身。

**建議**：新增 `docs/metrics.md` 追蹤：
```markdown
## 框架健康指標
- Spec 首次通過審核率：___% （目標 > 70%）
- 平均審核週期：___ 天 （目標 < 2 天）
- 衝突解決平均時間：___ 天
- 實作成功率（無需人工介入）：___% （目標 > 80%）
- 部署成功率：___% （目標 > 95%）
- 回饋迴路觸發次數/月：___
- 平均從 intake 到 deploy 的前置時間：___ 天
```

---

### 13. 多團隊協作支援

**問題**：框架假設單一團隊，沒有角色權限、spec 所有權、審核指派。

**建議**：
1. 在 spec template 加入所有權：
   ```markdown
   ## 負責人
   - Spec 擁有者：@name
   - 技術負責人：@name
   - 審核者：@name
   - 審核期限：YYYY-MM-DD
   ```

2. 在 `AGENTS.md` 加入審核 SLA：
   ```
   - 審核必須在 spec 提交後 2 個工作天內完成
   - 超過期限自動升級到技術主管
   ```

---

### 14. 成功指標（Spec 層級）

**問題**：Spec 定義了「做什麼」但沒有定義「怎麼衡量成功」。

**其他工具怎麼做**：
- Kiro 的 spec 包含 "success metrics"
- 業界最佳實踐要求 KPI 定義

**建議**：在 spec template 加入：
```markdown
## 成功指標
- 上線後 30 天內，目標用戶使用率達到 ____%
- 相關支援工單減少 ____%
- 頁面載入時間 < ___ 秒
- 量測方式：[如何蒐集數據]
```

## P2 — 建議改進

### 15. 漸進式部署策略

**問題**：部署是全有或全無（all-or-nothing），沒有 canary 或漸進式部署。

**建議**：在 `/deploy` 中支援：
- Canary 部署（先導 5% 流量）
- 藍綠部署
- Feature flags 整合

---

### 16. 完整工作範例（End-to-End Example）

**問題**：框架文件齊全但沒有一個完整的範例走過全部 10 個階段。

**其他工具怎麼做**：
- Spec Kit 有完整的 tutorial repo 和日文教學
- Kiro 有 guided walkthrough

**建議**：新增 `docs/example-walkthrough.md`，包含：
1. 一個真實需求（如「新增搜尋功能」）
2. 走過全部 10 個階段的完整成品
3. 包含一個衝突偵測與解決的範例
4. 包含一次部署失敗觸發回饋迴路的範例

---

### 17. 非技術人員的存取方式

**問題**：框架聲稱「零門檻」，但非技術人員需要 clone git repo 並使用 Claude Code CLI。

**建議**：
- 短期：建立 GitHub Issue 模板，讓非技術人員透過 GitHub Web UI 提交需求
- 中期：建立簡單的 Web 表單 → 自動建立 intake 檔案
- 長期：整合 Slack/Teams bot，讓人在聊天中直接提需求

---

### 18. 國際化支援

**問題**：模板全部是繁體中文，非中文團隊無法使用。

**建議**：
- 模板改為雙語或可配置語言
- 新增 `config/language.yml` 控制模板語言
- 提供英文版模板作為替代

---

### 19. 工作量估算

**問題**：框架沒有任何估算機制（Story Points、T-shirt sizing）。

**建議**：在 plan template 加入：
```markdown
## 工作量估算
- 總體複雜度：S / M / L / XL
- 預估任務數：___
- 預估開發週期：___ 天
```

---

### 20. 事後檢討（Post-Mortem）

**問題**：部署失敗觸發回饋迴路，但沒有結構化的事後檢討流程。

**建議**：新增 `docs/postmortem-template.md`：
```markdown
# 事後檢討：[事件名稱]
## 時間線
## 根本原因
## 影響範圍
## 修復措施
## 預防措施
## 學到的教訓
```

## 與主流工具的差距總結

| 能力 | 此框架 | Spec Kit | Kiro | Tessl |
|------|--------|----------|------|-------|
| 調研/去重階段 | ❌ 無 | ✅ research.md | ✅ steering | ✅ 自動聚合 |
| 上下文壓縮 | ❌ 無 | ✅ 意圖壓縮 | ✅ 動態裁剪 | ⚠️ 部分 |
| 安全掃描 | ❌ 無 | ⚠️ 社群整合 | ✅ 內建 | ⚠️ 部分 |
| API 合約 | ❌ 無 | ✅ contracts/ | ✅ 內建 | ✅ 內建 |
| 測試分層 | ❌ 無 | ✅ Test-First | ✅ 分層 | ⚠️ 部分 |
| 並行任務 | ❌ 無 | ✅ [P] 標記 | ⚠️ 部分 | ✅ 自動 |
| Spec 版本 | ❌ 僅 git | ⚠️ 僅 git | ✅ 內建 | ⚠️ 部分 |
| 多 AI 後端 | ❌ 僅 Claude | ✅ 6+ 後端 | ✅ 自有 AI | ✅ 自有 AI |
| 多團隊協作 | ❌ 無 | ⚠️ 基本 | ✅ 內建 | ✅ 內建 |
| 衝突解決流程 | ❌ 只偵測 | ❌ 無此概念 | ❌ 無 | ❌ 無 |
| 閉環回饋 | ✅ 完整 | ❌ 無 | ⚠️ 部分 | ❌ 無 |
| 非技術人員友善 | ✅ 零門檻設計 | ❌ 開發者工具 | ❌ 開發者工具 | ❌ 開發者工具 |
| 部署整合 | ✅ 完整 CI/CD | ❌ 無 | ✅ 內建 | ⚠️ 部分 |
| 自動回滾 | ✅ 完整 | ❌ 無 | ✅ 內建 | ⚠️ 部分 |

### 此框架的獨特優勢（其他工具都沒有的）

1. **衝突偵測機制** — 目前市面上唯一主動偵測角色間需求衝突的框架
2. **閉環回饋** — 從監控告警自動回流到需求系統，形成完整閉環
3. **非技術人員零門檻** — 唯一面向全組織而非僅面向開發者的框架
4. **不可變 intake 歷史** — 原始需求作為不可篡改的審計記錄

**結論**：此框架在「閉環回饋」和「非技術人員友善」方面領先所有競品，但在「開發者工具鏈成熟度」（安全、測試、合約、上下文管理）方面落後。優先補齊 P0 缺口即可大幅拉近差距，同時保持獨特的競爭優勢。
