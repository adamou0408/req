# /setup - AI-Guided Infrastructure Initialization

## Description
透過對話引導使用者完成基礎設施初始化。使用者只需要一台裝好 Coolify 的伺服器和 API Token，其餘全部由 AI 透過 Coolify API 自動完成。

## Prerequisites
- 一台已安裝 Coolify 的伺服器（`curl -fsSL https://cdn.coollabs.io/coolify/install.sh | bash`）
- Coolify API Token（從 Coolify Dashboard > Settings > API Tokens 產生）

## Usage
```
/setup [--target=coolify]
```

## Behavior

### Phase 1：收集連線資訊
向使用者詢問：
1. **Coolify URL**：你的 Coolify 網址是什麼？（例如 `https://coolify.example.com`）
2. **API Token**：你的 Coolify API Token？

立即透過 `GET {url}/api/v1/health` 驗證連線。
再透過 `GET {url}/api/v1/version` 確認 API 可用。
再透過 `GET {url}/api/v1/servers` 確認 Token 權限。

如果失敗，診斷問題並引導使用者修正。

### Phase 2：環境探查
透過 API 自動探查現有資源：

```bash
# 查詢現有伺服器
GET /api/v1/servers

# 查詢現有專案
GET /api/v1/projects

# 查詢所有資源
GET /api/v1/resources
```

向使用者回報：
- 「你的 Coolify 上有 N 台伺服器、M 個專案、K 個資源」
- 「我建議建立以下結構：...」

### Phase 3：建立專案結構
根據當前 repo 的性質，透過 API 自動建立：

```bash
# 建立專案
POST /api/v1/projects
  { "name": "<repo-name>", "description": "Managed by req framework" }

# 建立 staging 環境
POST /api/v1/projects/{uuid}/environments
  { "name": "staging" }

# 建立 production 環境
POST /api/v1/projects/{uuid}/environments
  { "name": "production" }
```

### Phase 4：建立應用程式
根據 repo 中的 Dockerfile / docker-compose.yml，自動選擇建立方式：

```bash
# 若有 Dockerfile
POST /api/v1/applications/public
  {
    "project_uuid": "...",
    "environment_name": "staging",
    "server_uuid": "...",
    "git_repository": "https://github.com/{owner}/{repo}",
    "git_branch": "main",
    "build_pack": "dockerfile",
    "ports_exposes": "3000",
    "instant_deploy": false
  }

# 若有 docker-compose.yml
POST /api/v1/services
  { ... }
```

設定域名與 SSL：
```bash
PATCH /api/v1/applications/{uuid}
  { "fqdn": "https://staging.example.com" }
```

### Phase 5：建立所需資料庫與服務
根據 repo 的依賴分析（package.json、requirements.txt、docker-compose.yml 中的 services），自動建議並建立：

```bash
# 例如偵測到 PostgreSQL 依賴
POST /api/v1/databases/postgresql
  {
    "project_uuid": "...",
    "environment_name": "staging",
    "server_uuid": "...",
    "name": "<repo>-db-staging",
    "postgres_user": "app",
    "postgres_password": "<auto-generated>",
    "postgres_db": "<repo>_staging"
  }

# 例如偵測到 Redis 依賴
POST /api/v1/databases/redis
  { ... }
```

自動將資料庫連線資訊設為應用程式的環境變數：
```bash
POST /api/v1/applications/{uuid}/envs
  { "key": "DATABASE_URL", "value": "postgresql://...", "is_build_time": false }
```

### Phase 6：設定監控（選用）
詢問使用者是否需要監控：

```bash
# 建立 Prometheus + Grafana 服務
POST /api/v1/services
  {
    "type": "prometheus",
    "project_uuid": "...",
    "server_uuid": "...",
    ...
  }
```

### Phase 7：儲存設定
將所有建立的資源 UUID 和連線資訊儲存到本地設定：

```bash
# 寫入 infra/coolify/config.local.yml（已 gitignore）
coolify:
  url: "https://coolify.example.com"
  # token 不存檔，執行時從環境變數讀取
  
resources:
  project_uuid: "abc123"
  server_uuid: "def456"
  staging:
    app_uuid: "ghi789"
    db_uuid: "jkl012"
    redis_uuid: "mno345"
    domain: "staging.example.com"
  production:
    app_uuid: "pqr678"
    db_uuid: "stu901"
    redis_uuid: "vwx234"
    domain: "app.example.com"
```

同時引導使用者將 `COOLIFY_API_TOKEN` 加入 GitHub Secrets（用於 CI/CD）。

### Phase 8：驗證
執行完整驗證：
1. 啟動 staging 應用：`POST /api/v1/applications/{uuid}/start`
2. 等待部署完成：輪詢 `GET /api/v1/deployments/{uuid}`
3. 健康檢查
4. 回報結果

## 後續操作
初始化完成後，AI 可透過以下命令管理基礎設施：
- `/deploy` — 部署應用程式
- `/infra` — 管理基礎設施資源（新增 DB、擴展、查看狀態等）

## Constraints
- **MUST** 在每一步操作前向使用者說明即將執行的動作
- **MUST** 在每一步操作後回報結果
- **MUST** 將密碼和 Token 存在環境變數，不寫入 git 追蹤的檔案
- **MUST** 自動產生強密碼（資料庫等）
- **MUST** 如果操作失敗，診斷原因並提供修正建議
- **MUST NOT** 在沒有使用者確認的情況下刪除現有資源
- **MUST NOT** 將 API Token 寫入任何 git 追蹤的檔案
