## * 需求摘要

讓業務與行銷人員可在建立短網址時自行指定人類可讀的 slug（例如 `short.ly/spring-sale`），取代自動生成的隨機字串，提高客戶對連結的信任度。系統需在不影響平台整體穩定與品牌安全的前提下，支援此能力。

## * 使用者故事

### 角色 A：業務人員（End User）

- **作為** 業務人員，**我想要** 自行指定短網址的 slug 名稱，**以便** 讓客戶一眼看懂、提升點擊率。
- **驗收條件**：
  - [x] 可輸入 3~40 字元的 slug（僅限 `[a-z0-9-]`）
  - [x] 若 slug 已被占用，立即顯示「已被使用，請換一個」並建議替代名稱
  - [x] 建立成功後，下次可直接重用同一個 slug（自動帶入）
  - [x] 每分鐘最多 10 次建立請求（CONFLICT-001 解決結果 — Option 3 分層配額）

### 角色 B：平台管理員（Admin）

- **作為** 平台管理員，**我想要** 維護保留詞清單並監控建立速率，**以便** 避免品牌詞被搶或有人濫用資源。
- **驗收條件**：
  - [x] 可設定保留詞清單（命中則拒絕，回傳 `slug_reserved` 錯誤）
  - [x] 可查看每位使用者過去 24 小時的建立次數
  - [x] 單一帳號超過每分鐘 10 次時，自動進入 60 秒冷卻期

## * 非功能需求

- 效能：建立 slug 的 P95 延遲 < 300 ms（含 uniqueness 檢查）
- 相容性：既有自動產生的 slug 行為不變，新功能為 opt-in

## * 安全性需求

- 資料分類：內部（含使用者識別）
- 認證需求：登入使用者才能建立自訂 slug
- 授權需求：Admin 可編輯保留詞清單；End User 僅能建立自己帳號下的 slug
- 加密需求：傳輸中 TLS；資料庫靜態加密依平台現行政策
- 審計日誌：記錄每次建立與拒絕事件（包括 reason: slug_taken / slug_reserved / rate_limited）
- 個資處理：slug 不應含 PII，系統不對此做內容過濾（後續 spec 可處理）

## * 成功指標

- 目標 1：上線後 30 天內，自訂 slug 功能的使用率 > 40%
- 目標 2：`slug_taken` 錯誤觸發率 < 5%（代表命名衝突罕見）
- 量測方式：產品分析工具 + 平台日誌
